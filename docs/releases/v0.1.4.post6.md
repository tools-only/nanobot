# v0.1.4.post6

Release date: 2026-04-11

## Summary

This release refines the layered knowledge architecture so low-level collection and high-level synthesis are no longer conflated.

## Highlights

- Xiaohongshu ingest now defaults to low-level archival outputs only:
  - `raw/xiaohongshu/`
  - `parsed/xiaohongshu/`
  - `canonical/archive/xiaohongshu/`
- High-level synthesis is now explicit:
  - queue at ingest with `--queue-expansion`
  - or promote an existing note with `nanobot knowledge expand enqueue-note <path>`
  - materialize promoted notes into `synthesis/fusion/`
- Knowledge artifacts now carry explicit semantics:
  - `kind`
  - `status`
  - `claim_type`
  - `derived_from`
  - `related_notes`
  - `confidence`
- Filesystem knowledge retrieval now understands existing YAML-frontmatter notes across the whole vault instead of only parsing runtime-emitted JSON-like notes under strict layer roots.

## User-Facing Changes

- `nanobot knowledge status` now reports `auto_queue_on_ingest`
- `nanobot knowledge xhs collect-url "<url>" --queue-expansion`
- `nanobot knowledge xhs scan-topic "<topic>" --queue-expansion`
- `nanobot knowledge expand enqueue-note <vault-relative-path>`

## Migration Notes

- Existing manual vault notes remain readable by the runtime.
- Existing archive notes can be promoted into synthesis later without re-ingest, as long as they exist in the vault and have usable frontmatter or links.
- Default behavior is intentionally conservative: ingest does not auto-synthesize unless explicitly requested.

## Verification

- Python compile checks passed for updated knowledge, CLI, config, and gateway modules.
- Runtime smoke checks confirmed:
  - low-level Xiaohongshu ingest writes `raw` / `parsed` / `canonical/archive`
  - default ingest does not create expansion jobs
  - explicit expansion writes `synthesis/fusion`
  - legacy YAML notes remain queryable
